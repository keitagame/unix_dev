================================================================================ 第4章 スケジューラ ================================================================================ 【4-1. スケジューラの役割】 複数のプロセスが「同時に」動いているように見せる仕組み。 実際はCPU時間を細かく分けて各プロセスに割り当てる（タイムスライス）。 いつ誰に切り替えるかを決めるのがスケジューリングアルゴリズム。 【4-2. CFS（Completely Fair Scheduler）の概念】 Linux 2.6.23以降の主要スケジューリングアルゴリズム。 コアアイデア: 「すべてのプロセスが等しいCPU時間を受け取る理想的なプロセッサ」をシミュレート。 実際には各プロセスが実行されるたびに「仮想時間（vruntime）」を進める。 常に vruntime が最も小さい（最も損をしている）プロセスを次に実行する。 vruntime の計算: delta_exec = 実際に実行した時間（ナノ秒） weight = nice値から決まる重み（nice 0 = 1024, nice 1 = 820, ...） delta_vruntime = delta_exec * NICE_0_WEIGHT / weight → 重みが大きい（nice値が小さい＝高優先度）ほど vruntime の増分が小さい → vruntime が小さいままになるので次回選ばれやすい データ構造: ランキューごとに「赤黒木（rb-tree）」を持つ vruntimeでソートされた木構造 最左ノード（vruntime最小）が次の実行候補（O(1)でアクセス可） 【4-3. ランキューの構造】 CPUごとに1つのランキュー（rq）を持つ（SMP対応）: struct rq { unsigned int nr_running; // 実行可能プロセス数 struct cfs_rq cfs; // CFSのランキュー struct rt_rq rt; // リアルタイムランキュー struct task_struct *curr; // 現在実行中のタスク struct task_struct *idle; // アイドルタスク u64 clock; // ランキューのクロック（ナノ秒） // ... }; スケジューリングクラスの優先順位: RT（リアルタイム）クラス > CFS クラス > Idle クラス RTタスクがある限りCFSタスクは実行されない 【4-4. コンテキストスイッチの実装】 schedule() → context_switch(rq, prev, next) の流れ: [1] switch_mm(prev->mm, next->mm, next) → CR3 に next->mm の PML4 アドレスをロード → TLB を無効化（ただし ASID/PCIDが使える場合は不要） [2] switch_to(prev, next, prev) → 汎用レジスタを保存（スタックに push） → prev の RSP を prev->thread.sp に保存 → RSP に next->thread.sp をロード → TSS（Task State Segment）の RSP0 を next のカーネルスタックに更新 ※ RSP0: ユーザー→カーネル遷移時に使うカーネルスタックポインタ → next の RSP からレジスタを復元（pop） → RET 命令で next のコードに飛ぶ （next のスタックにはリターンアドレスが積まれている） アセンブリレベルのイメージ（x86-64）: ; --- prev を保存 --- push rbp push rbx push r12 push r13 push r14 push r15 mov [prev + THREAD_SP], rsp ; --- next に切り替え --- mov rsp, [next + THREAD_SP] pop r15 pop r14 pop r13 pop r12 pop rbx pop rbp ret ; → next の実行コンテキストに戻る 【4-5. タイマー割り込みとプリエンプション】 タイマー割り込み（HZ = 100〜1000回/秒）のたびに: [1] rq->clock を更新 [2] 現在タスクの se.sum_exec_runtime を更新 [3] scheduler_tick() を呼ぶ [4] CFSは vruntime を更新し、最左ノードより大きすぎたら need_resched フラグをセット need_resched フラグが立ったら: → 割り込みからカーネルに戻る際に schedule() を呼ぶ（プリエンプション） カーネルプリエンプション: 2.6以降のLinuxはカーネル内でも安全な場所（preempt_count == 0）でプリエンプション可能 スピンロック取得中は preempt_count++ でプリエンプション禁止 【4-6. SMP（マルチコア）対応：ロードバランシング】 ロードバランサは定期的（またはコアがアイドルになったとき）に実行: [1] 各CPUのランキューの負荷（ランニングタスク数 × 重み）を計算 [2] 最も忙しいCPUから最も暇なCPUへタスクを移動（migrate） [3] タスクのマイグレーション: prev CPU のランキューから外し、next CPU に追加 キャッシュ親和性（cache affinity）考慮: 直前にそのCPUで動いていたタスクは同じCPUに残した方がキャッシュヒット率が高い →負荷差が一定以上にならないとマイグレーションしない ================================================================================ 第5章 システムコール機構 ================================================================================ 【5-1. システムコールとは】 ユーザー空間（リング3）からカーネル（リング0）のサービスを呼ぶ仕組み。 直接関数呼び出しではなく、CPUの特権遷移命令を使う。 x86-64 では SYSCALL 命令を使う（32bit時代は INT 0x80）: SYSCALL: ユーザー→カーネルへの高速遷移 SYSRET: カーネル→ユーザーへの高速帰還 【5-2. IDT（割り込みディスクリプタテーブル）】 IDT は 256 エントリのテーブル（ベクタ番号 0〜255）。 IDTR レジスタ（LIDT命令でロード）がベースアドレスとサイズを保持。 各エントリ（ゲートディスクリプタ）は 16バイト: struct idt_entry { uint16_t offset_low; // ハンドラアドレスの下位16bit uint16_t selector; // コードセグメントセレクタ uint8_t ist; // IST（割り込みスタックテーブル）インデックス uint8_t type_attr; // タイプ・DPL・Pビット uint16_t offset_mid; // ハンドラアドレスの中位16bit uint32_t offset_high; // ハンドラアドレスの上位32bit uint32_t reserved; // 0 } __attribute__((packed)); type_attr の内訳: bit7: P（Present）= 1 bit6-5: DPL（0=カーネルのみ呼べる、3=ユーザーから INT n で呼べる） bit4: 0（ゲートの場合） bit3-0: 0b1110 = 64bit割り込みゲート（IFビット自動クリア） 0b1111 = 64bit トラップゲート（IFビットそのまま） よく使う割り込みベクタ番号: 0: #DE ゼロ除算 1: #DB デバッグ 2: NMI（ノンマスカブル割り込み） 3: #BP ブレークポイント（INT3） 4: #OF オーバーフロー 6: #UD 無効オペコード 8: #DF ダブルフォルト 13: #GP 一般保護違反 14: #PF ページフォルト（CR2にフォルトアドレス） 32-255: 外部デバイス割り込み（ベクタはPIC/APICで設定） 【6-3. 割り込み発生時のCPU動作】 割り込み/例外が発生すると CPUは自動で: [1] 現在の SS と RSP をスタックに積む（ユーザー→カーネル遷移時のみ） [2] RFLAGS を積む [3] CS と RIP を積む [4] エラーコードを積む（#PF, #GP 等一部の例外のみ） [5] IDT[ベクタ番号] からハンドラアドレスと CS を読む [6] CS:RIP をハンドラに変更してジャンプ ハンドラの責務: [1] 残りのレジスタを保存（汎用レジスタ等）→ pt_regs 構造体 [2] C 言語のハンドラ関数を呼ぶ [3] レジスタを復元 [4] IRET 命令（割り込みからの帰還） → CS, RIP, RFLAGS を復元 → ユーザー空間なら SS, RSP も復元 【6-4. PIC と APIC の設定】 レガシー PIC（8259A）: 2枚のPIC（マスタ+スレーブ）でIRQ 0〜15 を管理 マスタPIC: ポート 0x20（コマンド）, 0x21（データ） スレーブPIC: ポート 0xA0（コマンド）, 0xA1（データ） 初期化（ICWシーケンス）後、IRQをベクタ番号32〜47にマップ 使用後は EOI（End of Interrupt）コマンドを送る（0x20に0x20を書く） APIC（Advanced PIC）: モダンなシステムはLocal APIC（各CPU）+ I/O APIC を使う Local APICはメモリマップドI/O（アドレス 0xFEE00000 等） I/O APICはIRQをLocal APICのベクタにルーティング SMP（マルチコア）に対応、IPI（プロセッサ間割り込み）も使える タイマー割り込みの設定（PIT: プログラマブルインターバルタイマー）: PIT ポート: 0x40（CH0 カウンタ）, 0x43（コントロール） クロック周波数 1193182 Hz を HZ（例:1000）で割った値をロード → 1ms ごとに IRQ0 が発生 【6-5. 割り込みハンドラの実装パターン】 割り込みハンドラは2段構成が一般的（デバイスドライバで多用）: 上半分（top half / hard IRQ ハンドラ）: → 割り込みコンテキストで実行（プロセスコンテキストではない） → 最小限の処理のみ（データを受け取る、フラグをセットする） → スリープしてはいけない → 可能な限り短く終わらせる 下半分（bottom half）: → 後で遅延実行（割り込みが戻ってから） → softirq, tasklet, work queue のいずれかを使う softirq: → 静的に定義された32個のソフト割り込み → TIMER_SOFTIRQ, NET_RX_SOFTIRQ, NET_TX_SOFTIRQ 等 → do_softirq() で処理（割り込みハンドラ終了後、スケジューラ前等） tasklet: → softirq の上に構築された動的な遅延実行機構 → 同一taskletは同時に1CPUでのみ実行される保証あり work queue: → カーネルスレッドで実行されるため、スリープできる → kworker スレッドが処理する ================================================================================ 第7章 仮想ファイルシステム（VFS） ================================================================================ 【7-1.
